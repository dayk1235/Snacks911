# Phase 2.3 — Roles & Audit

## Goal
Implementar seguridad estricta mediante RLS (Row Level Security), sistema de auditoría automatizado y herramientas de exportación de datos para control administrativo.

## Tasks

  <objective>DB Migration — Audit Logs System</objective>
  <read_first>
    - "supabase/migrations/20260411_initial_schema.sql"
    - "supabase/migrations/20260411_employee_auth.sql"
  </read_first>
  <action>
    Crear `supabase/migrations/20260428_audit_logs.sql`.
    1. Crear tabla `audit_logs` con campos: `id`, `table_name`, `record_id`, `action` (INSERT/UPDATE/DELETE), `old_data` (JSONB), `new_data` (JSONB), `changed_by` (TEXT), `created_at`.
    2. Crear función `process_audit_log()` en PL/pgSQL que capture el cambio y lo inserte en `audit_logs`.

<task>
  <objective>DB Migration — Restrict RLS Policies</objective>
  <read_first>
    - "supabase/migrations/20260411_initial_schema.sql"
  </read_first>
  <action>
    Crear `supabase/migrations/20260428_restrict_rls.sql`.
    1. Revocar permisos de escritura `ALL` a `anon` en `products` y `business_settings`.
    2. Actualizar políticas de `orders`: `anon` solo puede hacer `INSERT`. Solo `authenticated` (o admins verificados) pueden hacer `UPDATE` o `DELETE`.
    3. Implementar chequeo de rol en las políticas de escritura: `EXISTS (SELECT 1 FROM employees WHERE employee_id = ... AND role = 'admin')`.
  </action>
  <acceptance_criteria>
    - `anon` ya no tiene permiso `UPDATE` en la tabla `products`.
    - La tabla `orders` permite `INSERT` público pero restringe `UPDATE` a personal autorizado.
  </acceptance_criteria>
</task>
