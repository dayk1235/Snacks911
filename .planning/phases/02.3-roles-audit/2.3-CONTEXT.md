# Phase 2.3: Roles & Audit — Context

**Gathered:** 2026-04-28
**Status:** Ready for planning

<domain>
## Phase Boundary
Esta fase se enfoca en la seguridad y trazabilidad del sistema. Debemos pasar de una base de datos abierta (`anon` con permisos totales) a un sistema basado en roles (`admin` vs `staff`) y registrar cada cambio crítico para auditoría.

### Objetivos
1. Restringir el acceso a la base de datos mediante RLS (Row Level Security).
2. Implementar un sistema de logs de auditoría automatizado en Postgres.
3. Crear interfaces administrativas para ver logs y exportar datos.
</domain>

<decisions>
## Implementation Decisions

### Roles y Permisos
- **Admin**: Acceso total a todas las tablas y funciones. Único rol que puede gestionar empleados, precios y configuraciones de negocio.
- **Staff (Empleado)**:
    - Lectura: Productos, Órdenes, Caja, Configuración (solo lectura).
    - Escritura: Crear órdenes, actualizar estado de órdenes, movimientos de caja.
    - Bloqueado: No puede editar precios de productos, no puede borrar logs, no puede ver el dashboard de ventas totales (solo sus ventas del día).

### Sistema de Auditoría
- Tabla `audit_logs` en esquema `public`.
- Triggers de Postgres para capturar automáticamente INSERT/UPDATE/DELETE en:
    - `products` (cambios de precio/disponibilidad)
    - `orders` (cambios de estado/total)
    - `business_settings` (estado de tienda/número WA)
    - `employees` (creación/edición de usuarios)
- Los logs deben incluir el `old_value` y `new_value` en formato JSONB.

### Exportación de Datos
- Formato: CSV.
- Ubicación: Nueva pestaña "Reportes" en el Dashboard Admin.
- Entidades exportables: Órdenes (por rango de fechas) y Logs de Auditoría.
</decisions>
